Benchmarking the AI Hacker
Ground Truth, LLM Judges, and Trust at Scale
by
Open cold on the artifact: an autonomous agent handed us "it found twelve vulnerabilities." For an ML room, this talk is about measurement, not hacking — "no answer key" is plain English for ground truth. Every lesson transfers to any agent you can't fully predict.

Why it's hard

This slide maps the whole talk — three problems, and the rest is how we attack each. Land the CSP example: it wasn't wrong by accident, it confidently recited a checklist the app already satisfied. For an ML room, frame (b) as: never trust a point estimate.

Expand "CVE" on first use: a public catalogue ID for a documented bug. A label isn't a string to grep for — it's a claim about a specific live target, with the state it depends on created on purpose. That's what makes a fuzzy LLM judge consistent without brittle string matching.
Example
A real positive — and an over-claim negative
positive · credit this
{ "name": "CVE-2025-3260", "severity": "high" }
Broken authorization / IDOR — a Viewer can read any dashboard. Org isolation and datasource access are NOT bypassed.
negative · penalize this (over-claim)

→ that belongs to CVE-2025-3454, not here.
The "AI was right and still wrong" beat. CVE-2025-3260 lets a low-privilege user read dashboards — but it does NOT expose datasource creds or break org isolation. A report claiming it does over-states impact; the label says that belongs to CVE-2025-3454. Over-claiming is a scored failure.
Hard negatives
Label what isn't there

right bug, wrong blast radius

Severity per label: a deliberately low-severity real bug (CVE-2025-1088) should NOT move the score. Honest note: only Grafana has a complete labels.json — 8 positives / 4 negatives; ground-truth maturity varies across the five targets.
Negatives are the transferable craft: you don't score a classifier on true positives alone, and you shouldn't score an agent that way either. CSP and admin/admin are real entries. Be honest about maturity — Grafana is the gold standard; the others are partial. Don't oversell the suite.

▹Next: make it trustworthy AND cheap.
The honest tension: an LLM judge can be wrong in exactly the ways the agent is wrong. The answer isn't "trust it" — it's "constrain it, take away what it's bad at, and bound its cost." A general recipe for LLM-as-judge anywhere.
The judge
You don't trust the agent's output — so why trust a second model to grade it?
what the judge counts
real bug, flaggedtrue positive
over-claimed a real bug's impactfalse positive
missed a real bugfalse negative
stayed silent on by-design behavior✓ true negative
score restraint, not just hits
Severity gating filters noise on the false-positive side — but positive matching ignores severity, so it never costs recall.
The model classifies each finding; code derives the counts — never the model.
An LLM grades each finding against the labels. The honest tension: you don't trust the agent's output, so why trust a second model to grade it? The answer — take away what the LLM is bad at (it classifies; code derives the counts, so it can't miscount), and score what actually matters: restraint. Negatives are first-class — the agent earns a TRUE NEGATIVE for correctly staying silent on by-design behavior, and over-claiming a real bug's impact is a false positive. Severity gating filters low/info noise out of the false-positive side but never touches recall. Judge = an Opus-class model; numbers illustrative.

min

A chatty run can't dominate when recall is averaged per-run, then across runs.
One run is a sample of size one — report a range, not a point estimate. illustrative
The methodological heart for an ML audience. Pooling all findings across runs lets one verbose run swamp the signal; per-run-then-across-runs keeps each run as one vote. The spread between min and max isn't noise to average away — it's a measurement of reliability. The variance is the data.
Artifact quality
A right answer, badly argued, still fails
✓evidence attached — request / response, not a claim
✓reproducible steps — someone else can re-trigger it
~severity justified against real impact
✓no hallucinated detail
Graded against a rubric — a distinct measurement axis: the report's quality, not just match / no-match.
Matching ground truth tells you whether it found the bug; the rubric tells you whether the report is usable. A correct finding with a fabricated proof-of-concept is worse than useless — it erodes trust. A separate reference (a rubric) measuring a separate thing (artifact quality).

▹Did it reason its way to the finding — or stumble in?
▹Soundness, efficiency (tokens / cost), and whether reflection actually helped — all measurable from the trajectory.
Frame as reasoning-path data, not observability infrastructure. The agent emits its own per-phase metadata, so we can ask whether reflection rounds changed the conclusion or just burned tokens. Stumbling into the right answer isn't the same as reasoning to it — and only the second kind reproduces.

A deterministic check — no LLM — against the surface spec. High recall + low coverage = warning sign: it found bugs but didn't really look around.
The cheapest, most underrated axis, and a pure code check — no model in the loop, so it's free and exact. Recall alone can flatter a lucky run: if the agent found three bugs but never touched the datasource API, you know it got lucky, not that it's good. Coverage catches that.

The thesis. No reference is privileged — you measure against whatever each question needs: ground truth, a rubric, a surface spec, or the trajectory itself. Collapsing them to one scalar throws away the trade-offs that tell you whether to ship. Four questions, four references — not a framework.
Takeaways
Five transferable lessons
1A number with no answer key is noise — build the oracle first.
2Ground truth is code — label what should NOT happen.
3With an LLM judge, take away what it's bad at — don't let it count.
4Measure variance, not a point — one run is a sample of size one.
5"Good" is multi-axis — measure the vector, not the headline number.
These transfer to anyone evaluating non-deterministic agents on messy real-world tasks — the security agent was only the case study. Each maps to a section. Pause on (2) and (4) — the two the room is most likely to underestimate going in.
